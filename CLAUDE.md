# Genshin Tarot Server — правила для Claude Code

## Тесты карт (`tests/cards/`)

### Структура файла

Каждый тест-файл начинается с JSDoc-комментария с описанием карты:

```ts
/**
 * НазваниеКарты — что делает карта (эффект, цель).
 * Тип: Attack / Skill / Other. Стоимость: N очков действия.
 */
```

`describe` повторяет это описание: `"НазваниеКарты — что делает карта"`.

В `beforeEach` мок-сервер **всегда инициализируется конкретным игровым состоянием** — не пустым. Это значит: карта добавлена в руку, враги расставлены в нужную зону, AP и HP выставлены явно через `adminSetStats`. Тест не должен зависеть от дефолтных значений, которые могут измениться.

---

### Перед написанием тестов

Прочитай реализацию карты (`src/storage/cards/...`) и выпиши все входные параметры, которые она принимает из `CardUseContext`:
- `enemies` — передаётся? обязательно? сколько используется?
- `isUseAlternative` — есть ли альтернативный режим?
- `selectedPlayer`, `selectedCard` — используются?

Для каждой атаки в карте зафиксируй:
- `isPiercing` — пронзающая или нет?
- `isRange` — дальняя или нет?
Я хочу, чтобы ты проверял разные способы использования карты, чтобы потом можно было по тесту понять, какой запрос нужно отправить на сервер, чтобы повторить такое поведение.

Для каждого параметра, поведение которого **не очевидно из кода** (например: что происходит при `isUseAlternative`, если у карты нет явной ветки для этого случая; что происходит, если передан враг с 0 HP), **спроси пользователя** перед написанием теста. Не придумывай логику самостоятельно.

---

### Покрытие сценариев

Тесты должны охватывать **все варианты входных данных**, которые клиент может прислать:

- Основной сценарий (враг выживает)
- Враг погибает от удара
- Щит блокирует урон (если карта атакует)
- `isUseAlternative: true` — если карта имеет альтернативный режим
- Несколько врагов в `enemies` — проверить, что карта бьёт правильного
- Отсутствие `enemies` / пустой массив — если карта требует цель
- Любые другие обязательные поля, отсутствие которых должно давать ошибку

---

### Что проверять в каждом тест-кейсе

**Наличие** каждого ожидаемого шага — через `findIndex` с условием по `type` и идентификатору:
```ts
const idx = steps.findIndex((s) => s.type === EDetailedStep.X && s.field === value);
expect(idx).toBeGreaterThanOrEqual(0);
```

**Содержимое** каждого найденного шага — через `toEqual`, со всеми полями:
```ts
expect(steps[damageIdx]).toEqual({
  type: EDetailedStep.EnemyTakeDamage,
  enemyId: enemy.ID,
  damage: 3,
  isPiercing: false,
});
```
Проверка `findIndex >= 0` не считается достаточной — она не ловит неверные числа и флаги.

**Порядок** шагов там, где он важен для анимации на UI:
```ts
expect(damageIdx).toBeLessThan(deathIdx);
```

**Итоговое состояние** объектов после применения карты:
```ts
expect(enemy.Health).toBe(4);
```

**Стандартные шаги** (`PlayerStatChange` для actionPoints, `MoveCard` с `to: "discard"`) проверяются **в каждом тест-кейсе**, не только в первом. У `MoveCard` обязательно проверяется поле `card: { cardId, name, type }` — UI использует его, чтобы убрать конкретную карту из руки.

---

### Чего не делать

- Не проверять шаги, которые карта заведомо не генерирует в данном сценарии.
- Не добавлять `toHaveBeenCalledTimes(1)` в каждый тест — достаточно в одном (обычно первом).
- Не придумывать поведение карты для edge cases — лучше спросить.
